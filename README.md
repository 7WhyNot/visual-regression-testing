# VisionQA: Автоматизированная система визуального регрессионного тестирования

Данный проект является выпускной квалификационной (дипломной) работой на тему: **«Проектирование автоматизированной системы визуального регрессионного тестирования веб-интерфейсов»**.

В этом документе приведено детальное описание изначального технического задания, а также фактической реализации каждого пункта с описанием архитектуры и принятых инженерных решений.

---

## 1. Актуальность и цель работы

**Постановка задачи:**
Необходимость минимизации ручного труда QA-инженеров и предотвращения UI-дефектов при непрерывной интеграции (CI/CD). Цель — разработать платформу для автоматизированного попиксельного сравнения снимков веб-страниц с эталонными образцами и генерации отчетов о визуальных расхождениях.

**Фактическая реализация:**
Разработан полноценный SaaS-продукт (VisionQA), который позволяет создавать проекты, добавлять URL сценариев с указанием различных разрешений (Desktop, Mobile, Tablet) и в один клик запускать безголовый браузер для автоматического сбора скриншотов и их алгоритмического сравнения с эталонами. Реализован премиальный Enterprise UI интерфейс с поддержкой мультиязычности (i18n) и темной темы (Dark Mode) для комфортной работы QA-отдела.

## 2. Архитектура программного комплекса

### 2.1 Клиентская подсистема (Веб-интерфейс)
* **Планировалось:** Управление проектами, запуск тестов, визуализация отличий (наложение картинок).
* **Сделано:** Разработано полномасштабное SPA (Single Page Application) на базе React. 
  * Реализована навигация (Dashboard, Project Page, Run Report, Settings).
  * Интегрирован **DiffViewer** — продвинутый инструмент анализа скриншотов, который поддерживает 5 режимов просмотра: Baseline, Current, Swipe (Шторка с ползунком), Overlay (Наложение с цветовыми фильтрами: красный, неоновый розовый, желтый) и Side-by-Side (Рядом с синхронным скроллом). Поддерживается зумирование (ZoomIn/ZoomOut) и управление горячими клавишами.
  * Полный адаптивный дизайн с анимациями (`framer-motion`), поддержкой локализации (`react-i18next`) и переключением светлой/темной темы.

### 2.2 Серверная подсистема (Бэкенд)
* **Планировалось:** Хранение эталонов, маршрутизация, история прогонов.
* **Сделано:** Создан REST API на базе Node.js + Express.
  * База данных PostgreSQL управляется через ORM **Prisma**, что обеспечивает строгую типизацию и безопасность запросов. Созданы модели `Project`, `TestScenario`, `TestRun`, `TestResult`.
  * Интегрировано S3-совместимое хранилище **MinIO** для надежного и масштабируемого хранения бинарных файлов (сотен мегабайт скриншотов).

### 2.3 Подсистема автоматизации браузера (Worker)
* **Планировалось:** Изолированный модуль с Headless Browser для рендеринга страниц.
* **Сделано:** Реализован сервис `browser.service.js` на базе **Playwright**.
  * Поддерживает `fullPage` скриншоты.
  * Внедрены промышленные стандарты VRT: перед созданием снимка автоматически блокируются CSS-анимации, скрывается каретка ввода (курсор), замораживаются CSS-транзиции. 
  * Реализовано программное скрытие игнорируемых элементов (например, динамических баннеров или видео) через внедрение стилей `visibility: hidden !important`.

### 2.4 Модуль компьютерного зрения
* **Планировалось:** Сравнение изображений с учетом антиалиасинга и сдвигов.
* **Сделано:** Разработан модуль `compare.service.js` на основе библиотеки **pixelmatch** и **pngjs**.
  * Включен алгоритм компенсации антиалиасинга (`includeAA: false`) и задан порог чувствительности (`threshold: 0.1`), что минимизирует ложные срабатывания (Flaky tests).
  * Реализован **алгоритм нормализации размеров**: если старый и новый скриншоты отличаются по высоте (из-за динамического контента), система автоматически расширяет меньшее изображение прозрачными пикселями до размера большего, предотвращая падение алгоритма `pixelmatch` и корректно подсвечивая недостающие/добавленные блоки контента.

## 3. Требования к функциональным возможностям

| Требование ТЗ | Статус | Подробности реализации |
| :--- | :---: | :--- |
| **1. Создание тестовых сценариев** | ✅ | В интерфейсе проекта можно добавить сценарий (URL) и выбрать пресет устройства (Desktop 1920x1080, MacBook 1440x900, iPad, iPhone). |
| **2. Программный захват снимков** | ✅ | По кнопке "Запустить тесты" бэкенд открывает Playwright, проходит по всем сценариям проекта в фоне, ждет `networkidle` и делает снимки. |
| **3. Алгоритмическое сравнение и diff-изображение** | ✅ | Система автоматически создает Diff-изображение, где красным (или иным цветом) подсвечены измененные пиксели. Вычисляется точный процент расхождений. |
| **4. Ручная валидация оператором** | ✅ | На странице отчета (Run Report) QA-инженер видит список расхождений и может детально их изучить в DiffViewer. Доступны кнопки: "Утвердить эталон" (обновляет baseline в БД) и "Пометить как баг". |

## 4. Фактический технологический стек

* **Бэкенд:** Node.js, Express, Prisma ORM
* **Управление браузером:** Playwright (Chromium Headless)
* **Сравнение изображений:** pixelmatch, pngjs
* **База данных:** PostgreSQL
* **Хранилище файлов:** MinIO (S3 API)
* **Фронтенд:** React.js (Vite), Tailwind CSS, Framer Motion (анимации), Lucide React (иконки), React Router, react-i18next (Локализация RU/EN)

## 5. Инструкция по запуску (Development)

Проект состоит из двух частей: Backend и Frontend, а также использует Docker для инфраструктуры.

### 5.1 Запуск инфраструктуры (БД и S3)
Убедитесь, что установлен и запущен Docker.
```bash
cd backend
docker-compose up -d
```
Поднимутся два контейнера:
- **PostgreSQL** (порт 5432)
- **MinIO** (порт 9000 для API, 9001 для Console)

### 5.2 Запуск Бэкенда
```bash
cd backend
npm install
# Выполняем миграции БД (создание таблиц)
npx prisma migrate dev
# Устанавливаем браузеры для Playwright
npx playwright install chromium
# Запускаем сервер
npm start
```
Бэкенд запустится на `http://localhost:5000`.

### 5.3 Запуск Фронтенда
```bash
cd frontend
npm install
npm run dev
```
Фронтенд будет доступен по адресу `http://localhost:5173`.
